# Grad-Seminar Security Analysis of LLM Generated Rust Code

This project analyzes Rust code generated by large language models (ChatGPT, Gemini, Grok) in response to a real kernel CVE. The goal is to evaluate whether LLMs produce secure, idiomatic, and compilable Rust code when prompted to address **CVE-2025-68260** — the first Rust CVE in the Linux kernel, involving a race condition in the Android Binder driver.

## Overviewi

- Evaluate code quality and security across three prompting strategies: Zero-Shot, Chain-of-Thought, and Constraint-Based.
- Run each generated file through a pipeline: cargo project compile → CodeQL static analysis (general + CVE-specific) → unsafe block count.
- Produce structured CSV reports and charts comparing strategies.
- Perform manual peer review of all 30 samples to classify each as **Fixed**, **Vulnerable**, or **Inconclusive** with respect to CVE-2025-68260.

## How It Works

The analysis script (`analyze_samples_rsfiles.py`) is an assembly line that puts each Rust file through multiple inspection stations.

### The Setup: Finding the Code

Before analysis starts, the script scans your `Samples/` folder. It searches inside the specified subfolders (e.g., `chatgpt`, `gemini`, `grok`) for any `.rs` file. Each file is assigned a stable 12-character ID (SHA-256 hash of its path) so results never mix between runs.

### Station 2: Project Compile (Cargo)

Real Rust code needs a project context. This station builds a minimal temporary Cargo project around each file.

**Dependency inference:** The script scans each file for `use crate_name::` and `extern crate crate_name;` lines and auto-generates a `Cargo.toml` with those dependencies listed.

**`cargo check`** — Validates types, imports, and structure within a real project context.

**`cargo clippy`** — Rust's linter. Detects unidiomatic or unsafe patterns.

> **Note on `--clippy-deny-warnings`:** This flag treats every clippy warning as a hard failure. LLM-generated code consistently produces style warnings (e.g. `new-without-default`, `missing-safety-doc`) unrelated to security, resulting in 0% pass rate across all strategies. It is **not recommended** for this research — run clippy without it so that deny-by-default safety lints (e.g. `not_unsafe_ptr_arg_deref`) still fail the check, while cosmetic warnings do not.

**`cargo test`** (optional, `--run-tests`) — Executes any unit tests the LLM included in the file.

> **Note:** CodeQL (Station 3) only runs on files that pass `cargo check`. Files that fail are immediately marked `codeql=FAIL` with the reason `"CodeQL skipped: cargo check failed."` — running CodeQL on uncompilable code wastes time and produces no useful results.

### Station 3: Security Audit (CodeQL)

**What it does:** If enabled (`--run-codeql`), uses GitHub's CodeQL engine to perform deep static analysis. Two suites are run together in a single `codeql database analyze` call.

**How it works:**
1. Creates a fresh temporary Cargo project for the file.
2. Runs `codeql database create --language=rust --build-mode=none` — the Rust extractor indexes source directly without invoking cargo.
3. Runs `codeql database analyze` against all specified query suites with `--threads=1` to avoid memory pressure.
4. SARIF output is written to `output/{llm}/codeql/{hash}/codeql-results.sarif`.

**Recommended suites (pass both with `--codeql-suite`):**

| Suite | Purpose |
|---|---|
| `rust-security-extended.qls` | General CWE security alerts (CWE-022, CWE-089, CWE-362, etc.) |
| `codeql/rust-cve-2025-68260.qls` | Custom add-on: detects the specific bulk-transfer-under-lock race pattern from CVE-2025-68260 |

**Custom CVE-2025-68260 query (`codeql/CVE-2025-68260-RaceCondition.ql`):**

Detects the structural pattern where:
1. A collection is bulk-transferred via `mem::take` or `mem::replace` inside a lock-guarded block
2. The lock guard is dropped when that block exits
3. A `for` loop iterates the transferred collection after the lock is released

A concurrent `remove()` can corrupt list-node `prev`/`next` pointers between steps 2 and 3. Rule ID: `rust/lock-drop-before-list-traversal`.

> **Note:** The `codeql/qlpack.yml` file must exist alongside the `.ql` file. It declares the `codeql/rust-all` dependency so CodeQL can compile the custom query. This file is included in the repository.

**Prerequisites:**
- `codeql` CLI on `PATH` (CodeQL 2.25.1+ with Rust extractor support)
- At least one `--codeql-suite` flag pointing to a `.qls` file
- `--codeql-suite` may be repeated to run multiple suites in a single analysis pass

### Station 4: Unsafe Block Count

This station counts the number of `unsafe` blocks in each file using regex. `unsafe` usage is a necessary but not sufficient proxy for the CVE-2025-68260 pattern: all samples must use `unsafe` for intrusive pointer manipulation, so the count reflects complexity and how extensively the LLM reasoned about unsafe memory operations.

| Pattern | What it counts |
|---|---|
| `unsafe` | Unsafe blocks and function signatures |
| `extern "C"` | FFI calls to C |
| `ioctl`, `syscall`, `prctl` | Direct system calls |
| `binder`, `ashmem`, `ion` | Android memory management keywords |

Each hit records the line number and a 300-character snippet for manual review.

## Manual Peer Review

All 30 samples per LLM are reviewed manually to provide ground truth for CVE-2025-68260 detection. Each file is classified:

| Label | Pattern |
|---|---|
| **Fixed** | Pops nodes one-at-a-time while holding the lock, or holds lock across full traversal |
| **Vulnerable** | Bulk-transfers list via `mem::take`/`mem::replace` inside a lock scope, drops lock, then iterates |
| **Inconclusive** | No recognizable intrusive-list traversal, wrong abstraction, or ambiguous locking |

Manual review covers all 30 files regardless of whether they pass `cargo check` — the CVE pattern is algorithmic, not compilation-dependent. Results validate the custom CodeQL query and the `Stage4_Unsafe_Count` proxy metric.

## Usage

### Prerequisites

```bash
# Python environment
pip install pandas matplotlib seaborn

# Rust toolchain
rustup default stable

# CodeQL CLI (required only for --run-codeql)
# Download from: https://github.com/github/codeql-action/releases
# Place codeql binary on PATH and clone codeql repo:
#   git clone https://github.com/github/codeql codeql-home/codeql-repo
```

### Step 1 — Run the analysis pipeline

```bash
python3 analyze_samples_rsfiles.py --samples-root Samples/ --subfolders chatgpt --run-codeql --codeql-suite ~/codeql-home/codeql-repo/rust/ql/src/codeql-suites/rust-security-extended.qls --codeql-suite codeql/rust-cve-2025-68260.qls --output output/chatgpt

python3 analyze_samples_rsfiles.py --samples-root Samples/ --subfolders gemini --run-codeql --codeql-suite ~/codeql-home/codeql-repo/rust/ql/src/codeql-suites/rust-security-extended.qls --codeql-suite codeql/rust-cve-2025-68260.qls --output output/gemini

python3 analyze_samples_rsfiles.py --samples-root Samples/ --subfolders grok --run-codeql --codeql-suite ~/codeql-home/codeql-repo/rust/ql/src/codeql-suites/rust-security-extended.qls --codeql-suite codeql/rust-cve-2025-68260.qls --output output/grok
```

**Flag reference:**

| Flag | Description |
|---|---|
| `--samples-root Samples` | Root directory containing LLM subfolders |
| `--subfolders grok` | Comma-separated list of subfolders to scan (e.g. `chatgpt,gemini,grok`) |
| `--run-codeql` | Enable CodeQL static analysis (requires `codeql` on PATH) |
| `--codeql-suite <path>` | Path to a `.qls` suite file. **Repeat the flag** to run multiple suites in one pass |
| `--output output/grok` | Directory for all output files |
| `--timeout 900` | Per-command timeout in seconds (default: 900) |
| `--run-tests` | Run `cargo test` on each file (optional, slow) |
| `--clippy-deny-warnings` | Treat clippy warnings as failures. **Not recommended** — produces 0% pass rate for all strategies on LLM-generated code |

**Without CodeQL** (faster, for initial runs):
```bash
python3 analyze_samples_rsfiles.py \
  --samples-root Samples \
  --subfolders grok \
  --output output/grok
```

### Step 2 — Generate CSV reports and chart

```bash
python3 generate_report.py \
  --input output/grok \
  --output output/grok
```

### 2. Error Summary Report (`error_summary_report.csv`)

One row per file. Shows exactly why each file failed at each stage and what CodeQL found:

| Column | Meaning |
|---|---|
| `Strategy` | Prompting strategy |
| `File` | Path to the `.rs` source file |
| `S2 Check Error` | `cargo check` stderr output (if failed) |

| `S3 CodeQL Findings` | Comma-separated CWE rule IDs triggered, or `"none"` if CodeQL passed cleanly |

### 3. Performance Chart (`performance_chart.png`)

Two-subplot chart generated automatically by `generate_report.py`:

- **Top subplot** — Pass counts for Stage 2–3 (cargo check, clippy, CodeQL), Y-axis = number of samples passing. Colors: green (cargo check), orange (clippy), purple (CodeQL pass).
- **Bottom subplot** — Count metrics on their own Y-axis scale (counts are much larger than pass/fail values). Three bars per strategy: red (CodeQL security findings), dark blue (CVE-2025-68260 pattern hits), orange (unsafe block count).

## Key Design Decisions

**Stage 1 (rustc) removed from reports.** Direct `rustc` compilation on a single `.rs` file always fails (0/30 samples) because LLM-generated code uses kernel-style types and standard library imports that require a Cargo project context. Since this metric is flat zero for every strategy, it adds no information for comparing strategies or prompting techniques. It is still attempted internally but excluded from all aggregated output.

**CodeQL only runs on `cargo check`-passing files.** Attempting CodeQL on files with unresolvable dependencies causes hangs on network I/O. Since the Rust extractor uses `--build-mode=none`, dependencies are not fetched at the CodeQL step either — but dependency resolution errors at `cargo check` reliably indicate the file is not analyzable.

**`--build-mode=none` for CodeQL.** The CodeQL Rust extractor indexes source directly without invoking a build command. The `autobuild` mode is not supported for Rust as of CodeQL 2.25.1.

**`--threads=1` for CodeQL analyze.** Prevents the query evaluator from parallelizing into OOM on machines with limited RAM when running the security-extended suite.

**Multiple `--codeql-suite` flags.** The flag uses `action='append'` so it can be repeated. All suite paths are resolved to absolute paths before the `codeql database analyze` call, which runs with a temporary working directory — relative paths would fail without this resolution.

**`codeql/qlpack.yml` is required.** Without it, `codeql database analyze` cannot locate the `rust` dbscheme to compile `CVE-2025-68260-RaceCondition.ql` and exits non-zero. The `qlpack.yml` declares `codeql/rust-all: "*"` as a dependency, which resolves against the local `codeql-repo` checkout.

**`--clippy-deny-warnings` is not used.** With `-D warnings`, clippy fails on every LLM-generated file (0% pass rate, all strategies) due to cosmetic style warnings unrelated to security. Omitting the flag allows deny-by-default safety lints (e.g. `clippy::not_unsafe_ptr_arg_deref`) to still fail the check, which is the meaningful signal for this research.

**Dependency inference is heuristic.** The script infers `[dependencies]` from `use` and `extern crate` statements. It does not detect transitive dependencies or platform-specific crates (e.g., `kernel`, `android_binder`). Files using Linux kernel crates will fail `cargo check` because those crates are not available on crates.io.
