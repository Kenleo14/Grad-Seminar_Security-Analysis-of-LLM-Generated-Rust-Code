# Grad-Seminar Security Analysis of LLM Generated Rust Code

This project analyzes Rust code generated by large language models (ChatGPT, Gemini, Grok) in response to a real kernel CVE. The goal is to evaluate whether LLMs produce secure, idiomatic, and compilable Rust code when prompted to address **CVE-2025-68260** — the first Rust CVE in the Linux kernel, involving a race condition in the Android Binder driver.

## Overview

- Evaluate code quality and security across three prompting strategies: Zero-Shot, Chain-of-Thought, and Constraint-Based.
- Run each generated file through a four-station pipeline: raw compile → cargo project → CodeQL static analysis → CVE heuristic scan.
- Produce structured CSV reports and charts comparing LLMs and strategies.

---

## How It Works

The analysis script (`analyze_samples_rsfiles.py`) is an assembly line that puts each Rust file through multiple inspection stations.

### The Setup: Finding the Code

Before analysis starts, the script scans your `Samples/` folder. It searches inside the specified subfolders (e.g., `chatgpt`, `gemini`, `grok`) for any `.rs` file. Each file is assigned a stable 12-character ID (SHA-256 hash of its path) so results never mix between runs.

### Station 1: Raw Compile (rustc)

**What it does:** Tries to compile the `.rs` file directly using `rustc`.

This is a best-effort check — it doesn't care about dependencies. If raw syntax is broken, it fails here. The script notes the error and continues regardless.

**Behind the scenes:** Uses an absolute `-o` output path to avoid path-resolution errors across platforms.

### Station 2: Project Compile (Cargo)

Real Rust code needs a project context. This station builds a minimal temporary Cargo project around each file.

**Dependency inference:** The script scans each file for `use crate_name::` and `extern crate crate_name;` lines and auto-generates a `Cargo.toml` with those dependencies listed.

**`cargo check`** – Validates types, imports, and structure within a real project context.

**`cargo clippy`** – Rust's linter. Detects unidiomatic or unsafe patterns. With `--clippy-deny-warnings`, any warning becomes a hard failure.

**`cargo test`** (optional, `--run-tests`) – Executes any unit tests the LLM included in the file.

> **Note:** CodeQL (Station 3) only runs on files that pass `cargo check`. Files that fail here are immediately marked `codeql=FAIL` with the reason `"CodeQL skipped: cargo check failed."` — running CodeQL on uncompilable code wastes time and produces no useful results.

### Station 3: Security Audit (CodeQL)

**What it does:** If enabled (`--run-codeql`), uses GitHub's CodeQL engine to perform deep static analysis against known vulnerability patterns.

**How it works:**
1. Creates a fresh temporary Cargo project for the file.
2. Runs `codeql database create --language=rust --build-mode=none` — the Rust extractor indexes source directly without invoking cargo.
3. Runs `codeql database analyze` against the specified query suite with `--threads=1` to avoid memory pressure.
4. SARIF output is written to `output/{llm}/codeql/{hash}/codeql-results.sarif`.

**Recommended query suite:** `rust-security-extended.qls` (security-only queries). The full `rust-security-and-quality.qls` suite includes expensive quality queries that can cause memory exhaustion on small files.

**Prerequisites:**
- `codeql` CLI on `PATH` (CodeQL 2.25.1+ with Rust extractor support)
- `--codeql-suite` flag pointing to a `.qls` file

### Station 4: CVE Heuristic Scan (CVE-2025-68260)

This station is tailored to the Android Binder vulnerability research. It scans the raw source text for red-flag patterns using regex:

| Pattern | What it catches |
|---|---|
| `unsafe` | Unsafe blocks |
| `extern "C"` | FFI calls to C |
| `ioctl`, `syscall`, `prctl` | Direct system calls |
| `netlink`, `bpf`, `perf_event_open` | Kernel-level APIs |
| `/dev/...` | Device file paths |
| `libc::`, `use libc` | libc bindings |
| `nix::`, `use nix` | nix crate usage |
| `binder`, `ashmem`, `ion` | Android memory management |

Each hit records the line number and a 300-character snippet for manual review.

## Usage

### Prerequisites

```bash
# Python environment
pip install pandas matplotlib seaborn

# Rust toolchain
rustup default stable

# CodeQL CLI (required only for --run-codeql)
# Download from: https://github.com/github/codeql-action/releases
# Place codeql binary on PATH and clone codeql repo:
#   git clone https://github.com/github/codeql codeql-home/codeql-repo
```

### Step 1 — Run the analysis pipeline

```bash
python3 analyze_samples_rsfiles.py \
  --samples-root Samples \
  --subfolders grok \
  --run-tests \
  --clippy-deny-warnings \
  --run-codeql \
  --codeql-suite ~/codeql-home/codeql-repo/rust/ql/src/codeql-suites/rust-security-extended.qls \
  --output output/grok/
```

**Flag reference:**

| Flag | Description |
|---|---|
| `--samples-root Samples` | Root directory containing LLM subfolders |
| `--subfolders grok` | Comma-separated list of subfolders to scan (e.g. `chatgpt,gemini,grok`) |
| `--run-tests` | Run `cargo test` on each file |
| `--clippy-deny-warnings` | Treat clippy warnings as failures (`-D warnings`) |
| `--run-codeql` | Enable CodeQL static analysis (requires `codeql` on PATH) |
| `--codeql-suite <path>` | Path to a `.qls` query suite file |
| `--output output/grok/` | Directory for all output files |
| `--timeout 900` | Per-command timeout in seconds (default: 900) |

**Without CodeQL** (faster, for initial runs):
```bash
python3 analyze_samples_rsfiles.py \
  --samples-root Samples \
  --subfolders grok \
  --run-tests \
  --clippy-deny-warnings \
  --output output/grok/
```

### Step 2 — Generate CSV reports and chart

```bash
python3 generate_report.py \
  --input output/grok/ \
  --output output/grok/reports_csv/
```

Summary table of pass counts per prompting strategy:

| Strategy | Total_Samples | Stage1_rustc | Stage2_check | Stage2_clippy | Stage3_CodeQL_Pass | Stage3_CodeQL_Findings | Stage4_Unsafe_Count |
|---|---|---|---|---|---|---|---|
| Chain-of-Thought | 10 | 0 | 1 | 0 | 1 | 0 | 72 |
| Constraint-Based | 10 | 0 | 1 | 0 | 1 | 0 | 87 |
| Zero-Shot | 10 | 0 | 2 | 0 | 2 | 0 | 71 |

**Column descriptions:**

| Column | Meaning |
|---|---|
| `Total_Samples` | Number of `.rs` files analyzed |
| `Stage1_rustc` | Files that passed direct `rustc` compilation |
| `Stage2_check` | Files that passed `cargo check` |
| `Stage2_clippy` | Files that passed `cargo clippy` (strict mode) |
| `Stage3_CodeQL_Pass` | Files that CodeQL successfully analyzed |
| `Stage3_CodeQL_Findings` | Total CWE security alerts found across all CodeQL-analyzed files |
| `Stage4_Unsafe_Count` | Total number of `unsafe` blocks across all files per strategy |

### 2. Error Summary Report (`error_summary_report.csv`)

One row per file. Shows exactly why each file failed at each stage and what CodeQL found:

| Column | Meaning |
|---|---|
| `Strategy` | Prompting strategy |
| `File` | Path to the `.rs` source file |
| `S1 Error` | `rustc` stderr output (if failed) |
| `S2 Check Error` | `cargo check` stderr output (if failed) |
| `S2 Clippy Issue` | `cargo clippy` stderr output (if failed) |
| `S3 CodeQL Issue` | Reason CodeQL was skipped or failed (e.g. `"CodeQL skipped: cargo check failed."`) |
| `S3 CodeQL Findings` | Comma-separated list of CWE rule IDs triggered, or `"none"` if CodeQL passed cleanly |
| `S4 Heuristic Hits` | CVE-related keyword counts per pattern (if any hits found) |

### 3. Performance Chart (`performance_chart.png`)

Two-subplot chart generated automatically by `generate_report.py`:

- **Top subplot** — Pass counts for Stage 1–3 (compilation and CodeQL), Y-axis = number of samples passing
- **Bottom subplot** — Count metrics: CodeQL security findings and total `unsafe` block count per strategy, on their own Y-axis scale (unsafe counts are much larger than pass/fail counts and cannot share the same axis)

## Key Design Decisions

**CodeQL only runs on `cargo check`-passing files.** Attempting CodeQL on files with unresolvable dependencies causes `cargo fetch` to hang on network I/O. Since the Rust extractor uses `--build-mode=none`, dependencies are not fetched at the CodeQL step either — but dependency resolution errors at the `cargo check` stage are a reliable indicator that the file is not analyzable.

**`--build-mode=none` for CodeQL.** The CodeQL Rust extractor indexes source directly without invoking a build command. The `autobuild` mode is not supported for Rust as of CodeQL 2.25.1.

**`--threads=1` for CodeQL analyze.** Prevents the query evaluator from parallelizing into OOM on machines with limited RAM when running against the security-extended suite.

**Dependency inference is heuristic.** The script infers `[dependencies]` from `use` and `extern crate` statements. It does not detect transitive dependencies or platform-specific crates (e.g., `kernel`, `android_binder`). Files using Linux kernel crates will fail `cargo check` because those crates are not available on crates.io.
