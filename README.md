# Grad-Seminar Security Analysis of LLM Generated Rust Code

This project focuses on analyzing Rust code generated by large language models (LLMs) such as ChatGPT, Gemini, and Grok. The goal is to evaluate the security implications and performance of the generated code in real-world scenarios.

## Overview

- Study the security aspects of code generated by LLMs.
- Compare the generated code with manually written code for efficiency and safety.
- Create a structured repository to facilitate the analysis.

## Getting Started

To get started with this project, clone the repository and install any required libraries as specified in the accompanying documentation.

## How It Works

The analysis script is like an assembly line that puts each Rust file through multiple inspection stations. Here's what happens at each stage:

### The Setup: Finding the Code

Before the analysis starts, the script looks in your `Samples/` folder. By default, it hunts inside the `chatgpt`, `gemini`, and `grok` subfolders for any file ending in `.rs`. It gives each file a unique 12-character ID (a hash) so it doesn't mix up results.

### Station 1: The Quick Compile (rustc)

**What it does:** It tries to compile the Rust file directly using the basic `rustc` command.

**The Analogy:** Imagine asking a student, "Can you read this single sentence out loud without stuttering?" It doesn't care about the broader context; it just wants to see if the raw syntax is valid enough to create a binary executable.

**Behind the scenes:** It explicitly outputs the compiled file to an absolute path in your `out/` folder to prevent Windows/Linux path errors. If it fails, the script notes the error but keeps going.

### Station 2: The Deep Inspection (Cargo)

Rust code rarely lives in a vacuum; it usually needs a project structure (a `Cargo.toml` file) and external libraries (dependencies) to work.

**The "Fake" Project:** The script reads the `.rs` file and looks for lines like `use nix::` or `extern crate libc;`. It magically generates a temporary, hidden Cargo project just for that one file, writing a `Cargo.toml` that includes those necessary dependencies.

**cargo check:** It checks if the code is structurally sound and types align perfectly within a full project context.

**cargo clippy:** This is Rust's strict grammar checker. It looks for unidiomatic code (code that works, but is written poorly or unsafely). You can even set it to fail if it finds any warnings.

**cargo test (recommendation):** If you pass the `--run-tests` flag, it will run any internal unit tests the LLM might have generated inside the file.

### Station 3: The Security Audit (CodeQL)

**What it does:** If you enable it (`--run-codeql`), this step uses GitHub's CodeQL engine to perform advanced static analysis.

**The Analogy:** This is like sending the essay to a specialized plagiarism and deep-logic checker. It builds a database out of the Rust code and queries it for known vulnerability patterns (like memory leaks or data races).

### Station 4: The Heuristic Scan (Hunting CVE-2025-68260)

This is the most custom part of your script, tailored specifically to your graduate research on the Android Binder vulnerability.

**What it does:** It scans the raw text of the code looking for specific "red flag" words related to the CVE.

**The Red Flags:** It uses Regular Expressions (Regex) to search for:
- `unsafe` – Did the LLM use unsafe blocks?
- `ioctl`, `syscall` – Is it making direct system calls?
- `binder`, `ashmem` – Is it talking to Android's memory management?
- `extern "C"` – Is it trying to talk to C code?

**The Output:** It records exactly which line the word appeared on and grabs a 300-character snippet of that line for you to review manually later.

### Example Scenario

Let's say the script picks up `Samples/grok/zeroShot_Gr/cveZERO_1.rs`.

1. **Station 1:** It runs `rustc cveZERO_1.rs`. Let's say it succeeds. (Status: OK)

2. **Station 2:** It notices the file uses `libc`. It builds a temporary folder, creates a `Cargo.toml` with `libc = "*"`, and runs `cargo clippy`. Clippy finds a warning about a poorly named variable. (Status: FAIL if strict mode is on).

3. **Station 3:** You didn't run with `--run-codeql`, so it skips this. (Status: SKIP)

4. **Station 4:** It scans the text and finds the word `unsafe` on line 42, and `binder` on line 15. It saves these lines.

**The Final Report:** It bundles all this info into a neat `report.json` file inside your `out/reports/...` folder. Once all files are done, it spits out a combined `combined_report.json` and a quick terminal summary.

### Key Folders & Files

**`Samples/`** – The input data  
Contains Rust code samples generated by three LLMs (ChatGPT, Gemini, Grok), each tested with three prompting strategies:
- **Chain-of-Thought:** LLM given step-by-step reasoning guidance
- **Constraint-Based:** LLM given specific security/safety constraints
- **Zero-Shot:** LLM given minimal context, generating code directly

Each `.rs` file is a single code sample responding to the CVE-2025-68260 Android Binder vulnerability prompt.

**`output/`** – The results  
Organized by LLM, containing:
- **`reports/`** – Individual JSON files for each analyzed sample with detailed results
- **`combined_report.json`** – Aggregated statistics across all samples for that LLM
- **Performance charts** – PNG visualizations comparing results by prompt strategy

**`analyze_samples_rsfiles.py`** – The core analysis engine  
Runs each Rust file through the four-station inspection pipeline (rustc → cargo → CodeQL → heuristic scan).

**`generate_report.py`** – Report & visualization generator  
Processes all individual reports and creates summary statistics and charts.

## Usage

## instruction:

### Python environment

This project use simple python environment with pandas, matplotlib and seaborn

### Experiment running:
```bash
python3 analyze_samples_rsfiles.py --samples-root Samples --subfolders grok --run-tests --clippy-deny-warnings --output output/grok/
```

### Command Breakdown

This specific command executes a rigorous automated evaluation of the Rust code generated by the LLMs. Here is exactly what each flag in your command is doing:

- **samples-root Samples**: Tells the script where the base directory of your dataset is located.

- **subfolders**: Narrows the search to only look inside the subdirectory

- **run-tests**: Instructs the tool to execute cargo test on every file, checking if the LLM wrote functional, passing unit tests.

- **clippy-deny-warnings**: Enables "strict mode" for code quality. It passes the -D warnings flag to cargo clippy, meaning any stylistic or unidiomatic code will be treated as a hard failure.

- **output output/**: Directs all generated binaries, temporary files, and the final JSON reports into a dedicated output folder.

### Report Examples

Run this after running analyzsis experiment

```bash
python3 generate_report.py --input output/grok/ --output output/grok/
```
### Command Breakdown

This specific command executes

 **output output/{LLM_name}**: Directs all generated report and visualization to
  **input output/{LLM_name}**: Input the report from analysis experiments

After running the analysis scripts, you'll get structured reports. Here's what to expect:

### 1. Performance Report ( save to /output/{LLM}/performance_report.csv)

The `generate_report.py` script produces a summary table showing how many samples passed or failed at each stage for each prompt strategy:

| Strategy | Total_Samples | Stage1_rustc | Stage2_check | Stage2_clippy | Stage3_CodeQL | Stage4_CVE_Heuristic |
|----------|---------------|--------------|--------------|---------------|---------------|----------------------|
| Chain-of-Thought | 10 | 0 | 0 | 0 | 0 | 0 |
| Constraint-Based | 10 | 0 | 10 | 0 | 0 | 0 |
| Zero-Shot | 10 | 0 | 10 | 0 | 0 | 0 |

**Reading the columns:**
- **Total_Samples** – How many `.rs` files were analyzed
- **Stage1_rustc** – Files that failed basic `rustc` compilation
- **Stage2_check** – Files that failed `cargo check` structural validation
- **Stage2_clippy** – Files that failed `cargo clippy` idiomaticity checks
- **Stage3_CodeQL** – Files flagged by CodeQL static analysis (if enabled)
- **Stage4_CVE_Heuristic** – Files containing red-flag keywords (unsafe, binder, ioctl, etc.)

### 2. Performance Chart (/output/{LLM}/performance_chart.png)

A visualization is automatically generated showing pass/fail rates across all stages and strategies:

![Performance Chart Example](output/grok/grok_performance_chart.png)

This chart makes it easy to compare which LLM and prompting strategy produces the most secure and idiomatic code.

### 3. Error Summary Report (/output/{LLM}/error_summary_report.csv)

For detailed debugging, each analysis run also generates an `error_summary_report.csv` that captures specific error messages at each stage:

| Strategy | S1 Error | S2 Check Error | S2 Clippy Issue | S3 CodeQL Issue | S4 Heuristic Hits |
|----------|----------|----------------|-----------------|-----------------|-------------------|
| Chain-of-Thought | `error: expected type, found ','` at cveCHAIN_4.rs:55 | Multiple unresolved module errors for `kernel` crate | N/A | N/A | Found `unsafe` on line 42, `binder` on line 15 |

**What each column tells you:**
- **S1 Error** – Syntax or compilation errors from `rustc`
- **S2 Check Error** – Type checking or structural errors from `cargo check`
- **S2 Clippy Issue** – Code quality warnings from clippy
- **S3 CodeQL Issue** – Advanced vulnerability patterns detected
- **S4 Heuristic Hits** – Lines containing CVE-related red flags with context snippets

This helps you quickly identify *why* a particular sample failed and what needs to be fixed.
